Recipe Search App

Project Background

Not the most imaginative idea, I acknowledge! But I wanted to find a project that would allow me to make a relatively complex app using Vue 3. My main goals were to make a single page app using Vue router, which would involve multiple views, passing data between them in a variety of ways (search, filter, favourite, etc.) I also decided to use Tailwind CSS for this project. It’s a library that I’d used here and there on a couple of sites before, but I wanted to see what it would be like to use as the basis for a whole app, and to experiment a bit more with customising the settings and optimising it into my build.

Outputs

I’m hosting the current version of the app on Netlify. Let me know if you find any recipes you like!

Project Description

It’s a fairly straightforward recipe search app. You search by creating a list of ingredients, and then it tries to find matches using the Spoonacular recipe API. Once you have the results, you can filter by ingredients, favourite, and view more information.

Process and Challenges

This was the first substantial project I’ve undertaken using Vue 3. Because it’s so new, there were a few challenges with the setup, and some of the coding, due to the relative lack of documentation and forum discussion of this version. But it was a worthwhile challenge and a helpful reminder of some of the costs of working with newer systems.

Aside from that, the main challenge was getting the overall architecture right, especially with regards to components. Sometimes it was obvious that I would be re-using a component many times (e.g. the recipe cards which show up on the results page and the favourites page). Other times I just refactored when I realised that I was repeating myself.

But there were a couple of other elements where it was more difficult to work out whether I should create a component or not, because whilst there was significant overlap between two uses, there were also some significant differences. For example, lists of ingredients show up pretty often in different forms throughout the app, and for a while I was thinking that I ought to have a dedicated component for this. However, once the way I was using the lists started to differentiate, it became clear that there wasn’t much to be gained by doing this, especially given that list rendering is so easy in Vue! So in the end I was glad that I didn’t jump to create a component first for that case, as it would have been premature and ultimately not too useful.

One interesting challenge I encountered with Tailwind was using it for layout. I love the way that Tailwind works for things like padding and margins, as having a set scale here makes it much easier to keep consistency, whilst also moving quickly. However, for things like width and height it is (or at least it seems to me so far) a little less straightforward. In fact, I ended up using grid quite heavily (e.g. within small elements) where I’d probably have just resized elements if I was writing my own classes. I don’t think that this ultimately matters too much in this context. However, given that (as I understand it) grid is relatively expensive in performance terms, this doesn’t seem ideal overall.

I don’t know yet whether this is something that’s really coming out of Tailwind, or if I just need to re-think how I’m using it. But I definitely plan to use it again so it’s something I’ll keep an eye on in the next project!
